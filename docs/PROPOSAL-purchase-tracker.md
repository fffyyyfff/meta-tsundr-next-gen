# 企画書: Meta-tsundr 購入管理プラットフォーム拡張

## エグゼクティブサマリー

Meta-tsundr を「積読管理」から **「あらゆるネット購入の統合管理プラットフォーム」** に進化させる。Gmail の購入確認メール（Amazon、楽天等）を AI エージェントが自動解析し、購入履歴を一元管理する。freee のクレジットカード同期のように、ユーザーが意識しなくても購入データが自動で蓄積される「アンビエント（環境型）エージェント」を実現する。

---

## 1. 市場調査・競合分析

### 1.1 既存サービスの課題

| サービス | 強み | 弱み |
|---------|------|------|
| **Gmail Purchases（2025年9月〜）** | Google公式、セットアップ不要 | APIアクセス不可、カスタマイズ不可、日本のEC対応不十分 |
| **Slice（Package Tracker）** | 2006年まで遡って購入履歴取得 | 開発停滞、日本のEC非対応 |
| **Rocket Money** | サブスク自動検出＋キャンセル代行 | 米国金融機関のみ、メール解析ではなく銀行API |
| **マネーフォワード/Moneytree** | 日本の金融機関2,400社連携 | 金額のみ（商品詳細なし）、メール非対応 |
| **freee** | クレジットカード明細自動同期 | 会計目的のみ、個人の購入管理ではない |
| **ブクログ** | 読書管理に特化 | メール連携なし、書籍以外非対応 |

### 1.2 市場の空白（Meta-tsundrが狙うポジション）

**誰もやっていないこと:**
- 日本のEC（楽天・Amazon.co.jp）のメールを **AI で自動解析** して購入リスト化
- **書籍に限らず全商品** を統合管理
- **アンビエント（常駐型）** で定期同期（freee のクレカ同期モデル）
- **AI による購入分析**（「今月の無駄遣い」「積読率」「同じ本の二重購入防止」）

## 2. ユーザーペインポイント（刺さるポイント）

### 2.1 プライマリペイン（強い課題）

| ペイン | ユーザーの声 | 解決策 |
|-------|-------------|--------|
| **「何買ったか覚えてない」** | Amazonの注文履歴は直近3ヶ月、楽天は確認しづらい | メールから全購入を自動リスト化 |
| **「同じ本を二重購入した」** | 紙で買ったのにKindleでも購入 | ISBN重複チェック + 購入済みアラート |
| **「月にいくら使ったか不明」** | サブスクの積み重ね（subscription creep） | カテゴリ別支出の自動集計 |
| **「返品期限を逃した」** | ECサイトごとに返品ポリシーが違う | 返品期限リマインダー |
| **「積読が増え続ける」** | 買うだけ買って読まない | 積読率の可視化 + AI 読書計画 |

### 2.2 セカンダリペイン（あると嬉しい）

| ペイン | 解決策 |
|-------|--------|
| 「発売予定の本を予約し忘れた」 | 楽天API + 発売日リマインダー |
| 「セール時に安く買えたか確認したい」 | 価格履歴追跡 |
| 「家族で同じ物を買ってしまった」 | 家族共有モード |
| 「確定申告で経費を整理したい」 | freee/MF連携エクスポート |

## 3. プロダクトビジョン

### 3.1 コンセプト: 「買い物の記憶を自動化する」

```
┌────────────────────────────────────────────────┐
│              Meta-tsundr                        │
│       「あなたの全購入を、覚えています」          │
│                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │  📚 書籍  │  │  📦 全商品 │  │  💳 支出  │      │
│  │  積読管理  │  │  購入履歴  │  │  分析     │      │
│  └─────┬────┘  └─────┬────┘  └─────┬────┘      │
│        │              │             │           │
│        └──────────────┼─────────────┘           │
│                       │                         │
│  ┌────────────────────┴──────────────────────┐  │
│  │        アンビエントエージェント            │  │
│  │  Gmail → AI解析 → 自動分類 → 通知         │  │
│  └───────────────────────────────────────────┘  │
└────────────────────────────────────────────────┘
```

### 3.2 差別化ポイント

1. **日本のEC特化**: 楽天・Amazon.co.jp・ヨドバシ・ZOZOTOWN のメールフォーマットに最適化
2. **AI 解析**: Claude による非構造化メールの自然言語解析（schema.org 非対応メールも処理可能）
3. **書籍 × 全商品の統合**: 積読管理の延長線上で全購入を管理
4. **アンビエント同期**: freee のクレカ同期のように、バックグラウンドで定期的にメールを取得
5. **プライバシーファースト**: メールデータはローカル/ユーザー所有のDBに保存、第三者に送信しない

## 4. 機能設計

### 4.1 Phase 1: Gmail 連携基盤（MVP）

| 機能 | 詳細 |
|------|------|
| **Gmail OAuth 連携** | `gmail.readonly` スコープで読み取り専用アクセス |
| **購入メール自動検出** | from: `auto-confirm@amazon.co.jp`, `order@rakuten.co.jp` 等でフィルター |
| **AI メール解析** | Claude API でメール本文から商品名、価格、数量、注文番号を抽出 |
| **購入リスト表示** | 日付順、カテゴリ別、EC別の表示 |
| **手動同期ボタン** | 「今すぐ同期」で最新メールを取得 |

### 4.2 Phase 2: アンビエント同期

| 機能 | 詳細 |
|------|------|
| **定期バックグラウンド同期** | 1時間ごとにGmailをポーリング（Google Pub/Sub push も検討） |
| **新規購入通知** | 新しい購入が検出されたらアプリ内通知 |
| **重複検出** | 同じISBN/商品名の二重購入を警告 |
| **カテゴリ自動分類** | AI が商品を「書籍」「家電」「食品」「衣類」等に自動分類 |
| **支出サマリー** | 日別/週別/月別の購入金額集計 |

### 4.3 Phase 3: インテリジェント機能

| 機能 | 詳細 |
|------|------|
| **返品期限リマインダー** | EC別の返品ポリシーに基づき通知 |
| **発売予定トラッキング** | 楽天API + 発売日アラート |
| **AI 購入分析レポート** | 「今月の傾向」「無駄遣い度」「積読率」 |
| **価格比較** | 同じ商品の楽天/Amazon価格比較 |
| **家族共有** | 家族メンバーの購入も統合表示 |
| **freee/MF エクスポート** | 確定申告用にCSV/API出力 |

## 5. 技術アーキテクチャ

### 5.1 メール取得・解析パイプライン

```
Gmail API (OAuth2, gmail.readonly)
  ↓
メールフィルター (from/subject パターンマッチ)
  ↓ 対象メールのみ
┌─────────────────────────────────┐
│ 二段階解析                       │
│                                  │
│ Step 1: schema.org/Order 検出    │
│   → 構造化データがあれば即抽出   │
│                                  │
│ Step 2: Claude AI 解析           │
│   → HTML本文を自然言語で解析    │
│   → 商品名、価格、数量を抽出   │
└────────────┬────────────────────┘
             ↓
購入データ正規化 (Purchase モデルに変換)
             ↓
DB 保存 + 重複チェック + カテゴリ分類
             ↓
UI 表示 + 通知
```

### 5.2 対応ECサイト（優先順）

| EC | メール差出人 | 件名パターン | 難易度 |
|----|-------------|-------------|--------|
| **Amazon.co.jp** | `auto-confirm@amazon.co.jp` | 「ご注文の確認」 | 中（HTML複雑） |
| **楽天市場** | `order@rakuten.co.jp` | 「注文内容ご確認」 | 中 |
| **楽天ブックス** | `order@books.rakuten.co.jp` | 「ご注文確認」 | 低（書籍特化） |
| **ヨドバシ.com** | `*@yodobashi.com` | 「ご注文ありがとうございます」 | 低 |
| **ZOZOTOWN** | `*@zozo.com` | 「ご注文確認」 | 低 |
| **メルカリ** | `*@mercari.jp` | 「購入完了」 | 低 |
| **Kindle** | `*@amazon.co.jp` | 「Kindle本のご注文」 | 低 |

### 5.3 データモデル

```
Purchase (購入)
  ├── id
  ├── userId
  ├── source: "amazon" | "rakuten" | "yodobashi" | ...
  ├── orderNumber
  ├── orderDate
  ├── items[]
  │   ├── name
  │   ├── price
  │   ├── quantity
  │   ├── category: "book" | "electronics" | "food" | ...
  │   ├── imageUrl
  │   ├── isbn (書籍の場合)
  │   └── productUrl
  ├── totalAmount
  ├── currency
  ├── status: "ordered" | "shipped" | "delivered" | "returned"
  ├── returnDeadline
  ├── emailId (Gmail message ID)
  ├── rawEmailSnippet
  └── syncedAt
```

### 5.4 既存アーキテクチャとの統合

```
Browser ──tRPC──→ Next.js BFF
                    ├── book.* (既存の積読管理)
                    ├── purchase.* (新規: 購入管理)
                    │   ├── purchase.sync (Gmail同期)
                    │   ├── purchase.list (購入一覧)
                    │   ├── purchase.stats (支出分析)
                    │   └── purchase.detect (重複検出)
                    ├── gmail.* (新規: Gmail連携)

                    │   ├── gmail.sync (手動同期)
```

## 6. セキュリティ・プライバシー設計

### 6.1 データ保護方針

| 項目 | 方針 |
|------|------|
| **メールアクセス** | `gmail.readonly` のみ（送信・削除不可） |
| **データ保存** | ユーザーのDBにのみ保存、第三者に送信しない |
| **メール本文** | 購入データ抽出後は本文を破棄（スニペットのみ保持） |
| **AI 解析** | Claude API に送信するのは購入メールのみ（全メール送信しない） |
| **OAuth トークン** | 暗号化して保存、いつでも revoke 可能 |
| **GDPR/個人情報保護法** | データエクスポート・完全削除機能を提供 |

### 6.2 Sift の教訓

Sift（旧Sift Wallet）はメールデータを第三者に販売して問題になった。Meta-tsundr は **データ販売を一切行わない** ことを明確にし、プライバシーポリシーで宣言する。

## 10. リスクと対策

| リスク | 影響 | 対策 |
|-------|------|------|
| Gmail API の利用制限 | 同期頻度制限 | Pub/Sub push 通知、バッチ処理 |
| ECメールフォーマット変更 | パーサー破損 | AI解析フォールバック、モニタリング |
| プライバシー懸念 | ユーザー離脱 | 透明性、データ最小化、ローカルファースト |
| Claude API コスト | 運用費増大 | キャッシュ、ローカルLLM検討 |
| Google OAuth 審査 | リリース遅延 | 早期に審査申請、スコープ最小化 |
